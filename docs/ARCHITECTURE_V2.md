# RAG Funding Engine — Architecture V2 (Generic Multi-Schedule Platform)

> **Version:** 2.0  
> **Status:** Architecture Specification (Migration from V1 ACC1520-Only System)

---

## 1. Overview

The RAG Funding Engine has evolved from a single-purpose ACC1520 billing assistant into a **generic multi-schedule platform**. Instead of hardcoded logic for a single schedule, the system now dynamically profiles any uploaded schedule document and generates schedule-specific queries and business rules.

### Key Evolution

| Aspect | V1 (Legacy) | V2 (Current) |
|--------|-------------|--------------|
| **Schedule Support** | Single ACC1520 schedule only | Multiple schedules, dynamically profiled |
| **Code Logic** | Hardcoded ACC1520 heuristics | LLM-generated schedule profiles |
| **Query Generation** | Static patterns | Dynamic patterns from document analysis |
| **Storage** | Single SQLite DB | Isolated per-schedule databases |
| **API** | Implicit schedule context | Explicit `schedule_id` parameter |

---

## 2. Key Concepts

### 2.1 Schedule Profile

A **Schedule Profile** is dynamically generated by LLM analysis of the uploaded PDF. It serves as the "brain" for that specific schedule, containing:

```json
{
  "schedule_type": "rural_medical",
  "schedule_name": "Rural Practitioner Services Schedule",
  "key_dimensions": [
    "location_type",
    "travel_distance",
    "consultation_type",
    "after_hours"
  ],
  "unique_rules": [
    "Travel claims require distance > 20km from base",
    "After-hours premiums apply 18:00-08:00 and weekends",
    "Rural loading applies to locations classified DHB8-DHB12"
  ],
  "query_patterns": [
    "{specialty} consultation {location}",
    "travel claim {distance}km {location_type}",
    "after-hours {service_type} {time_range}",
    "rural loading {procedure_code}"
  ],
  "business_logic": {
    "has_travel_reimbursement": true,
    "has_after_hours_premiums": true,
    "has_location_tiers": true,
    "has_time_based_tiers": false,
    "requires_csc_check": false
  },
  "code_prefixes": {
    "GP": "General Practitioner services",
    "TR": "Travel reimbursement",
    "AH": "After-hours premium"
  }
}
```

**Storage Location:** `data/processed/{schedule_id}/manifest.json`

### 2.2 Dynamic Query Generation

Instead of hardcoded heuristics, query patterns are extracted from document content during ingestion:

**Example Schedule-Specific Patterns:**

| Schedule Type | Example Query Patterns |
|---------------|------------------------|
| **Rural Medical** | `{specialty} consultation {location}`, `travel claim {distance}km`, `rural loading {procedure}` |
| **Urgent Care** | `after-hours {service}`, `urgent consultation {urgency_level}`, `weekend premium {service}` |
| **GP Standard** | `standard consult {duration}`, `complex consultation {complexity_factors}`, `CSC {age_band}` |
| **Specialist** | `{specialty} procedure {complexity}`, `referral required {condition}`, `theatre time {duration}` |

### 2.3 Multi-Schedule Support

Each schedule is completely isolated with its own:

- **SQLite Database** — Codes, chunks, and embeddings
- **Schedule Profile** — LLM-generated manifest with business rules
- **Embeddings Store** — Vector index for semantic search
- **Version Tracking** — Source hash and ingestion metadata

**Router Logic:**
```python
# At query time, the router selects the correct schedule
schedule_id = request.schedule_id  # e.g., "acc1520-rural"
profile = load_profile(schedule_id)  # Load schedule-specific rules
db_path = f"data/processed/{schedule_id}/{schedule_id}.sqlite3"
```

## 4. Business Logic Discovery

The LLM discovers schedule-specific rules through a combination of keyword scanning and document classification:

### 4.1 Detection Methods

```python
# Keyword-based heuristics for initial classification
SCHEDULE_DETECTION_PATTERNS = {
    "rural": [
        "rural", "travel", "mileage", "kilometre", "km",
        "remote", "isolated", "on-call", "after-hours"
    ],
    "urgent_care": [
        "urgent", "after-hours", "weekend", "public holiday",
        "premium", "surcharge", "extended hours"
    ],
    "gp_standard": [
        "consultation", "CSC", "community services card",
        "standard", "long", "extended", "age"
    ],
    "specialist": [
        "specialist", "procedure", "theatre", "operating",
        "referral", "complex", "anaesthetic"
    ]
}
```

### 4.2 Schedule-Specific Behaviors

| Schedule Type | Key Business Rules | Example Code Selection |
|---------------|-------------------|------------------------|
| **Rural** | Location matters, mileage reimbursable, travel time billable | Same consult gets travel codes + location loading |
| **Urgent Care** | After-hours premiums, urgency modifiers, time-based surcharges | Emphasizes after-hours + urgency codes |
| **GP Standard** | Standard consults, time-based tiers, CSC eligibility | Emphasizes time + complexity + CSC status |
| **Specialist** | Procedure complexity, theatre time, referral requirements | Emphasizes procedure codes + complexity |

### 4.3 Example: Same Consult, Different Schedules

**Input:** `"Patient visit to remote area, 45km from clinic, evening consultation"`

| Schedule ID | Primary Codes | Reasoning |
|-------------|---------------|-----------|
| `acc1520-rural` | `GP1-RURAL`, `TR45`, `AH-EVENING` | Rural location + 45km travel + after-hours |
| `acc1520-urgent` | `UC-URGENT`, `AH-PREMIUM` | Urgent care context + after-hours surcharge |
| `acc1520-gp` | `GP-LONG` | Standard GP long consultation |

## 7. Migration Notes (V1 → V2)

### 7.1 API Changes

| V1 (Old) | V2 (New) |
|----------|----------|
| `ingest_acc1520(pdf)` | `ingest_schedule(pdf, schedule_id="acc1520-medical")` |
| `recommend_codes(db_path, ...)` | `recommend_codes(schedule_id, ...)` |
| Implicit single schedule | Explicit `schedule_id` parameter required |

### 7.2 Backward Compatibility

For existing ACC1520 deployments:

```python
# V1-style calls are automatically mapped
DEFAULT_SCHEDULE_ID = "acc1520-medical"

# Old endpoint behavior (no schedule_id specified)
if not request.schedule_id:
    request.schedule_id = DEFAULT_SCHEDULE_ID
```

### 7.3 Database Migration

```bash
# Migrate existing V1 database to V2 structure
mv data/processed/acc1520.sqlite3 data/processed/acc1520-medical/acc1520-medical.sqlite3

# Create manifest for existing schedule
cat > data/processed/acc1520-medical/manifest.json << 'EOF'
{
  "schedule_id": "acc1520-medical",
  "schedule_type": "gp_standard",
  "schedule_name": "ACC1520 Medical Practitioners and Nurses",
  "version": "ACC1520-v2",
  "profile": {
    "key_dimensions": ["consult_type", "age_band", "csc_status"],
    "business_logic": {
      "has_csc_tiers": true,
      "has_age_bands": true
    }
  }
}
EOF
```

### 7.4 Code Changes Required

**Before (V1):**
```python
from rag_funding_engine.pipeline.ingest_acc1520 import ingest_schedule
from rag_funding_engine.pipeline.recommend import recommend_codes

# Hardcoded paths
result = ingest_schedule(pdf_path=DEFAULT_PDF, out_dir=DEFAULT_OUT)
recommendations = recommend_codes(db_path=DEFAULT_DB, consult_text=text)
```

**After (V2):**
```python
from rag_funding_engine.pipeline.ingest import ingest_schedule
from rag_funding_engine.pipeline.recommend import recommend_codes

# Dynamic schedule routing
result = ingest_schedule(
    pdf_path=pdf_path,
    schedule_id="acc1520-rural",  # Required!
    out_dir=DEFAULT_OUT
)

recommendations = recommend_codes(
    schedule_id="acc1520-rural",  # Required!
    consult_text=text
)
```

## 10. Security & Isolation

### 10.1 Schedule Isolation

- Each schedule's data is completely isolated
- No cross-schedule queries possible without explicit intent
- Profile defines allowed operations per schedule type

### 10.2 Data Integrity

```python
# Source hash verification
if sha256_file(pdf_path) != manifest["source"]["source_hash"]:
    raise IntegrityError("Schedule source has changed")
```

## Appendix: Glossary

| Term | Definition |
|------|------------|
| **Schedule** | A billing/fee schedule document (PDF) containing codes and rules |
| **ScheduleProfile** | LLM-generated metadata describing a schedule's structure and rules |
| **Schedule ID** | Unique identifier for a schedule (e.g., `acc1520-rural`) |
| **Key Dimensions** | Variables that affect code selection (location, time, age, etc.) |
| **Query Patterns** | Templates for matching consult text to codes |
| **Business Logic** | Boolean flags indicating schedule capabilities |
| **Router** | Component that selects correct schedule DB/profile at runtime |
